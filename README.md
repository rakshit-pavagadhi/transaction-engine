## Overview

This project implements three core APIs — record a transaction, fetch a user's summary, and fetch a global ranking — backed by a relational schema (SQLite via SQLAlchemy), with a static HTML/JS frontend that exercises all three endpoints live.

The focus of the implementation is on idempotent writes, atomic updates, fair multi-factor ranking, and input validation — rather than on feature breadth.

## Tech Stack

| Layer | Choice |
|---|---|
| API framework | FastAPI |
| ORM / DB | SQLAlchemy + SQLite |
| Validation | Pydantic v2 |
| Frontend | Static HTML/CSS/vanilla JS (no build step) |
| Deployment | Vercel (`@vercel/python` + `@vercel/static`) |

SQLite was chosen for zero-setup local development and because Vercel's serverless filesystem (`/tmp`) supports it without external infrastructure. See [Assumptions & Trade-offs](#assumptions--trade-offs) for why this is **not** production-grade for concurrent writes, and what would replace it.

## Running Locally

**Prerequisites:** Python 3.9+

```bash
# 1. Clone and enter the repo
git clone <your-repo-url>
cd <repo-name>

# 2. Create a virtual environment
python -m venv venv
source venv/bin/activate      # Windows: venv\Scripts\activate

# 3. Install dependencies
pip install -r requirements.txt

# 4. Run the API
uvicorn api.index:app --reload --port 8000
```

The API will be available at `http://127.0.0.1:8000`. Interactive docs (Swagger UI) are auto-generated by FastAPI at `http://127.0.0.1:8000/docs`.

**Frontend:** open `public/index.html` directly in a browser, or serve the `public/` folder with any static file server. Ensure `API_BASE_URL` in `app.js` points at your running API.

## API Reference

All endpoints are prefixed with `/api` (matching the deployed routing in `vercel.json`).

### `POST /api/transaction`

Records a transaction for a user and updates their running totals and score.

**Request body:**
```json
{
  "user_id": "NXS-1042",
  "amount": 250.50,
  "idempotency_key": "a1b2c3d4-..."
}
```

| Field | Type | Constraints |
|---|---|---|
| `user_id` | string | required, min length 1 |
| `amount` | float | required, `0 < amount <= 100000` |
| `idempotency_key` | string | required, min length 1, must be unique per logical transaction |

**Response — `200 OK`:**
```json
{
  "transaction_id": "f47ac10b-...",
  "user_id": "NXS-1042",
  "amount": 250.50,
  "status": "success"
}
```

If the same `idempotency_key` is submitted again, the original transaction is returned unchanged with `status: "success (duplicate request prevented)"` — no new row is created and no totals are double-counted.

**Error responses:**
- `422 Unprocessable Entity` — validation failure (missing field, non-positive amount, amount over the cap)
- `500 Internal Server Error` — unexpected database error (logged server-side)

### `GET /api/summary/{user_id}`

Returns a user's current totals, score, and live rank.

**Response — `200 OK`:**
```json
{
  "user_id": "NXS-1042",
  "total_amount": 750.50,
  "transaction_count": 3,
  "score": 555.35,
  "rank": 2
}
```

**Error responses:**
- `404 Not Found` — no user with that `user_id` has any recorded transactions

### `GET /api/ranking?limit=10`

Returns the top N users ordered by score, descending.

| Query param | Type | Default | Notes |
|---|---|---|---|
| `limit` | int | 10 | Number of leaderboard entries to return |

**Response — `200 OK`:**
```json
{
  "rankings": [
    { "user_id": "NXS-1042", "total_amount": 750.50, "transaction_count": 3, "score": 555.35, "rank": 1 },
    { "user_id": "NXS-2031", "total_amount": 500.00, "transaction_count": 1, "score": 360.00, "rank": 2 }
  ]
}
```

## Data Model

Two tables:

**`users`**
| Column | Type | Notes |
|---|---|---|
| `user_id` | string, unique | external-facing identifier |
| `total_amount` | float | running sum of all transaction amounts |
| `transaction_count` | int | running count of all transactions |
| `score` | float | derived ranking score, recomputed on every transaction |
| `created_at` / `updated_at` | datetime | audit timestamps |

**`transactions`**
| Column | Type | Notes |
|---|---|---|
| `transaction_id` | string, unique | server-generated UUID, returned to the client |
| `user_id` | string | foreign reference to the user |
| `amount` | float | the transaction amount |
| `idempotency_key` | string, **unique** | client-supplied key; the uniqueness constraint is what makes duplicate prevention airtight at the DB level, not just at the application level |
| `timestamp` | datetime | server-side record time |

`total_amount`, `transaction_count`, and `score` on `users` are **denormalized aggregates** — they're recomputed and stored on every successful transaction rather than calculated on read. This keeps `GET /summary` and `GET /ranking` cheap (no aggregation query over the full `transactions` table), at the cost of needing to keep the aggregate in sync on every write — which is exactly what the single-transaction commit in `POST /transaction` is responsible for.

## Ranking Logic

Score is a weighted combination of two factors, recalculated on every transaction:

```
score = (total_amount * 0.7) + (transaction_count * 10)
```

This is intentionally **multi-factor**, rather than ranking by raw transaction volume alone:

- The `total_amount` term rewards users who move real value through the system.
- The `transaction_count` term gives a flat bonus per transaction, rewarding consistent/repeated engagement rather than a single large deposit.

**Why this resists trivial manipulation:** because both terms are weighted and additive, neither "one huge transaction" nor "many tiny transactions" alone dominates the score — a user has to combine both to rank highly. The `amount` field is also bounded (`0 < amount <= 100000` per transaction, enforced by Pydantic), which caps how much a single transaction can swing a score.

See [Known Limitations](#known-limitations) for where this formula could still be gamed, and what a production version would add.

## Duplicate & Concurrency Handling

**Duplicate prevention (idempotency):**
1. On each `POST /transaction`, the handler first checks if `idempotency_key` already exists in the `transactions` table. If found, it short-circuits and returns the original transaction's data with a `(duplicate request prevented)` status — no balance mutation occurs.
2. Because `idempotency_key` has a **database-level unique constraint** (not just an application-level check), two requests with the same key that race past the initial check at nearly the same instant will still only succeed once: the second insert raises `IntegrityError`, which is caught, rolled back, and resolved by re-querying for the now-existing row and returning it the same way.

This two-layer approach (pre-check + DB constraint + `IntegrityError` fallback) means duplicate prevention holds even under concurrent identical requests, not just sequential ones.

**Consistency:**
All mutations for a single transaction — creating/locating the user, updating `total_amount`/`transaction_count`/`score`, and inserting the new `Transaction` row — happen inside one SQLAlchemy session and are committed together with `db.commit()`. If any step fails, `db.rollback()` ensures no partial update is persisted (e.g. a transaction row without the corresponding balance update, or vice versa).

## Validation & Error Handling

- All request bodies are validated by Pydantic models before reaching business logic — malformed types, missing fields, and out-of-range amounts are rejected with `422` and a descriptive error body automatically generated by FastAPI.
- `amount` is bounded to `(0, 100000]` to reject non-positive amounts and cap single-transaction impact on the leaderboard.
- Lookups for non-existent users (`GET /summary/{user_id}`) return `404` rather than a generic error or empty success response.
- Unexpected exceptions during a transaction are caught, logged server-side, rolled back, and surfaced to the client as a generic `500` (to avoid leaking internal error detail to API consumers).

## Assumptions & Trade-offs

- **SQLite over Postgres/MySQL:** chosen for zero-config local setup and Vercel-compatible deployment without provisioning external infrastructure. This is explicitly a development/demo trade-off — see [Known Limitations](#known-limitations).
- **No authentication layer:** `user_id` is accepted as a free-form client-supplied string with no signup/login flow, since the assignment scope is the transaction/ranking engine itself, not identity management. In a real system, `user_id` would be derived from an authenticated session, not trusted as client input.
- **Idempotency key is client-generated:** the frontend generates a UUID v4 client-side as a sensible default; in production this would typically be required per logical operation (e.g., one key per "checkout") rather than regenerable on demand.
- **No transaction history endpoint:** only aggregate totals are exposed; per-transaction history retrieval was out of scope for the three required endpoints.

## Known Limitations

Documented honestly, as the assignment asks for trade-offs to be called out rather than hidden:

1. **Concurrent writes to the same user are not fully isolated.** SQLite combined with this code's read-then-write pattern (`user.total_amount = user.total_amount + request.amount`) is vulnerable to a lost-update race: two simultaneous transactions for the *same* `user_id` can both read the pre-update balance before either commits, causing one update to be overwritten. A production fix would use either:
   - A `SELECT ... FOR UPDATE` row lock (requires Postgres/MySQL, not SQLite), or
   - An atomic `UPDATE users SET total_amount = total_amount + :amount ...` statement instead of read-modify-write in Python, or
   - Optimistic locking via a version column.

   Duplicate-request prevention (the same key submitted twice) **is** race-safe, via the DB unique constraint — it's specifically *concurrent distinct transactions for one user* that share this risk.

2. **Ranking can still be gamed by sheer transaction volume.** Since each transaction adds a flat `+10` to the score regardless of size, a user could in theory submit many minimum-amount transactions to inflate `transaction_count` cheaply. A production version would add rate-limiting per user (e.g. max N transactions per minute) and/or a minimum-amount floor per transaction, or weight `transaction_count` logarithmically instead of linearly.

3. **No rate limiting or abuse detection at the API layer.** There's no per-IP or per-user request throttling; this would normally sit in front of the API (e.g. via a gateway or middleware) rather than in application code.

4. **SQLite on Vercel is ephemeral.** The deployed database lives in `/tmp`, which is wiped on cold starts/redeploys. The demo is stateless across deploys by design — this is acceptable for an evaluation demo but would not survive in a real multi-instance serverless deployment without an external database (Postgres on Neon/Supabase, PlanetScale, etc.).
